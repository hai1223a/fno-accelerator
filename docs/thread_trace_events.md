# Thread Trace 事件说明

`thread_trace.out` 用于按时间记录关键 SystemC 线程的粗粒度活动。它不是指令流水图，也不是 RTL 波形；它用来回答“某个仿真时刻哪些线程被唤醒、做了什么高层动作”。

## 文件格式

```text
# 时间ps 周期 线程 事件 说明
<时间ps> <周期> <线程> <事件> <说明>
```

- `时间ps`：当前 SystemC 时间，单位 ps。
- `周期`：`时间ps / (cycle_ns * 1000)`。
- `线程`：模型线程的中文名称。
- `事件`：该线程发生的高层动作。
- `说明`：事件相关的关键字段。

## 线程与事件

### IFU取指线程

| 事件 | 说明字段 | 含义 |
| --- | --- | --- |
| `线程唤醒` | `本周期PC=0x... 内部队列=... 有未处理响应=...` | IFU 主线程进入一轮处理；`本周期PC` 是本轮预计继续推进的取指 PC。 |
| `处理冲刷` | `目标PC=0x... 可发新请求=...` | EXU 本周期发来 flush，IFU 在同一仿真时间处理组合重定向并清空本地 packet。`可发新请求=1` 时本周期直接尝试访问目标 PC；`可发新请求=0` 时不会发旧路径请求。 |
| `记录延迟冲刷` | `目标PC=0x...` | flush 到来时已有 IFU request 在路上，IFU 先记住目标 PC，等待在途 response 返回后再补发目标 PC 取指。 |
| `处理延迟冲刷` | `目标PC=0x... 可发新请求=...` | 之前记录的 delayed flush 现在可以发出目标 PC 取指；通常紧跟在旧 request 的 response 被消费之后。 |
| `访问ITCM` | `PC=0x... lane地址=0x... 长度=8` | 当前取指 PC 不在 hold-up lane 中，IFU 发起真实 ITCM lane 读取；`lane地址` 是 64-bit 对齐访问地址，可能小于 PC。 |
| `接收lane` | `lane地址=0x... 长度=8 内容=0x...` | IFU 收到一条 64-bit ITCM lane；同一条 lane 即使拆出多条指令，也只记录一次接收。 |
| `提取ITCM指令` | `PC=0x... 指令=0x... lane地址=0x...` | IFU 从本周期刚返回的 ITCM lane 中直接提取一条指令；本周期不继续把同一条新 lane 的下一条指令记为命中。 |
| `命中lane` | `PC=0x... 指令=0x... lane地址=0x...` | 该指令来自上一轮已经缓存的 hold-up lane，没有触发新的 ITCM 访问。 |

### EXU执行线程

| 事件 | 说明字段 | 含义 |
| --- | --- | --- |
| `线程唤醒` | `mcycle=... 待冲刷=... 输入有效=... OITF占用=...` | EXU 每个 cycle 的主循环状态。 |
| `发出冲刷` | `目标PC=0x...` | EXU 将 commit/branch/trap 产生的重定向发送给 IFU；IFU 可能在同一时间戳记录 `处理冲刷`。 |
| `处理访存返回` | `接收LSU结果` | long-pipe LSU response 被 EXU 优先处理。 |
| `空闲等待` | `IFU输入无效` | EXU 当前没有可执行的 IFU packet。 |
| `派发执行` | `PC=0x... 指令=0x... 序号=...` | EXU 接收并执行当前指令。 |
| `暂停派发` | `PC=0x... 指令=0x... 序号=...` | 因 OITF/LSU/串行化条件不能 dispatch。 |

### LSU访存线程

| 事件 | 说明字段 | 含义 |
| --- | --- | --- |
| `线程唤醒` | `请求队列=... 未完成请求=...` | LSU 收到 EXU 请求或响应事件后被唤醒。 |
| `取出请求` | `PC=0x... 地址=0x... itag=... 读请求=...` | LSU 从内部队列取出 load/store 请求。 |
| `返回未对齐错误` | `地址=0x...` | LSU 检测到未对齐访问并直接返回错误。 |
| `等待响应` | `itag=...` | LSU 已发起 TLM 请求，等待 target 返回。 |
| `接收响应` | `PC=0x... 地址=0x... itag=... 错误=...` | DTCM/ITCM/BIU 返回 `BEGIN_RESP`，LSU 已经把它转换为 EXU 可消费的 long-pipe response。 |

### BIU转换线程

| 事件 | 说明字段 | 含义 |
| --- | --- | --- |
| `线程唤醒` | `有待处理请求=... 正在处理=...` | BIU 有新的 LSU->BIU non-blocking 请求需要转换。 |
| `转发阻塞访问` | `地址=0x...` | BIU 将 non-blocking 请求转换为下游 `b_transport`。 |
| `返回响应` | `地址=0x...` | 下游 blocking 访问完成，BIU 向 LSU 返回 `BEGIN_RESP`。 |

### ITCM响应线程

| 事件 | 说明字段 | 含义 |
| --- | --- | --- |
| `线程唤醒` | `有待处理请求=...` | ITCM 有 IFU 或 LSU 请求等待响应。 |
| `选择请求` | `来源=IFU/LSU 地址=0x...` | ITCM 仲裁并选择一个 pending 请求。 |
| `返回响应` | `来源=IFU/LSU 地址=0x...` | ITCM 完成 SRAM 访问并返回 `BEGIN_RESP`。 |

### DTCM响应线程

| 事件 | 说明字段 | 含义 |
| --- | --- | --- |
| `线程唤醒` | `有待处理请求=...` | DTCM 有 LSU 请求等待响应。 |
| `访问SRAM` | `地址=0x...` | DTCM 执行 SRAM 读写。 |
| `返回响应` | `地址=0x...` | DTCM 向 LSU 返回 `BEGIN_RESP`。 |

## 阅读建议

- 先看 `EXU执行线程 派发执行`，确认哪些指令真的进入执行。
- 再往前找同一个 PC 附近的 `IFU取指线程 提取ITCM指令/命中lane`、`接收lane`、`访问ITCM` 和 `ITCM响应线程 返回响应`，确认取指时序。
- 如果 EXU 出现 `空闲等待`，通常说明 IFU 没有提供有效 packet。
- 如果 EXU 出现 `发出冲刷`，同一时间戳应能看到 IFU 的 `处理冲刷`。如果 `可发新请求=1`，后面通常紧跟目标 PC 的 `访问ITCM`；如果 `可发新请求=0`，会看到 `记录延迟冲刷`，等旧 request 返回后再看到 `处理延迟冲刷`。
- 如果 LSU/BIU/DTCM/ITCM 出现等待链，可以按 `itag` 或地址串联访存请求和响应。
- DTCM/ITCM 的 `返回响应` 与 LSU 的 `接收响应` 通常在同一时间戳；EXU 会在同一周期优先记录 `处理访存返回`，下一周期恢复被 OITF 阻塞的年轻指令。

## Flush 时序

IFU flush 事件按 e203 RTL 的 `pipe_flush_req -> pc_nxt -> ifu_req_pc` 组合路径建模。典型可立即发请求的情况：

```text
40000 4 EXU执行线程 发出冲刷 目标PC=0x80000014
40000 4 IFU取指线程 处理冲刷 目标PC=0x80000014 可发新请求=1
40000 4 IFU取指线程 访问ITCM PC=0x80000014 lane地址=0x80000010 长度=8
```

如果 flush 到来时 IFU 仍有在途取指 request，则先记录延迟冲刷：

```text
40000 4 IFU取指线程 处理冲刷 目标PC=0x80000014 可发新请求=0
40000 4 IFU取指线程 记录延迟冲刷 目标PC=0x80000014
50000 5 IFU取指线程 处理延迟冲刷 目标PC=0x80000014 可发新请求=1
```
